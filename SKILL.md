# ppt-paperlogy-inspired-skill

마크다운(+이미지) → **편집 가능한 통일 디자인 .pptx**. 핵심 철학: **내용은 한 글자도 날조하지 않고, 디자인은 단일 시스템으로 강제한다.**

## 이 스킬의 성격 (필독)
- **전역 + 부트스트래핑**: 전역 스킬이다. 쓸 때는 메인 작업 폴더(`Projects/`) 하위에 `ppt-<주제>` 폴더를 만들고 거기서 부트스트랩해 동작한다(엔진 scripts/assets는 스킬에, 산출물은 작업 폴더에).
- **옵트인**: 자동으로 쓰지 마라. "이 문서를 ppt로 만들까요? (이 스킬 사용)"라고 **먼저 사용자에게 묻고** 동의할 때만 쓴다.
- **진화하는 디자인(기본값)**: 디자인은 고정이 아니다. 사용자 요청·피드백을 받을 때마다 `render.py`(디자인 시스템)를 조금씩 발전시킨다. **첫 PPT를 만드는 순간이 이 스킬이 "그 사용자의 스킬"로 길들여지기 시작하는 시점**이다. (통일성은 여전히 단일 렌더러로 보장 — 바뀌는 건 디자인 자체이지 통일 방식이 아니다)
- **템플릿 스킬 → 분화**: 이건 paperlogy에서 영감받은 **출발점 템플릿**이다. 사용자가 결과에 충분히 만족하는 시점이 오면 `paperlogy-inspired` 딱지를 떼고 **"이 스킬을 뭐라고 기억할까요?"** 라고 물어, 답한 이름으로 **새 스킬로 분화(포크)**시켜라.

## 아키텍처 (이걸 지켜야 통일성이 안 깨진다)
- **너(Claude)의 역할**: 원문을 읽고 → 사실을 보존하며 → 슬라이드 **스펙 JSON**을 만든다. (디자인은 절대 직접 그리지 않는다)
- **`scripts/render.py`의 역할**: 스펙 JSON을 **단 하나의 디자인 시스템**으로 렌더한다. 색·폰트·여백·프레임은 전부 여기서 고정된다.
- 너는 "무엇을 담을지", 렌더러는 "어떻게 보일지"를 담당한다. 이 분리가 통일성의 단일 원천이다.
- **절대 pptx XML을 직접 만들거나 python-pptx로 즉석 슬라이드를 그리지 마라.** 무조건 스펙 JSON → `build.py`.

## 워크플로

### 환경 준비 (자동 · 사용자에게 노출하지 말 것)
의존성/설치는 **사용자 블랙박스**다. 사용자에게 pip·python 설치를 시키지 마라. 빌드 전에 조용히 확인하고 없으면 설치한다:
```bash
python -c "import pptx, PIL" 2>NUL || pip install python-pptx pillow
```
(실패하면 그때만 사용자에게 알린다. 성공이면 언급하지 않는다.)

### 0. 온보딩 (config.json의 `onboarded`가 false일 때만, 1회)
처음 실행이면 PPT를 만들기 전에 사용자에게 묻고 답을 `config.json`에 저장한 뒤 `onboarded: true`로 바꾼다. (AskUserQuestion 사용)
1. **로고**: "우상단 로고를 어떻게 할까요? (기본 your-logo 유지 / 로고 PNG 경로 / 텍스트 로고 생성)" → 파일을 받으면 `assets/your-logo.png`를 교체하고 `python scripts/build_template.py`를 다시 돌려 마스터에 재임베드.
2. **포인트 컬러**: "강조색은 기본 블루(#1456F0)로 갈까요, 다른 브랜드 색이 있나요?" → 바꾸면 `config.json`의 `accent` 갱신.
3. **디자인 확인**: "회색 배경 + 포인트색 + Pretendard 미니멀 디자인 그대로 괜찮나요?" (나머지 스타일은 고정이라는 점 안내)
4. **폴더 구조 확인**: "주제마다 `decks/NN-슬러그/` 한 폴더에 md·이미지·결과 pptx가 함께 생기는 구조가 편한가요?"
온보딩을 건너뛰라고 하면 기본값으로 진행하고 `onboarded: true`로만 바꾼다.

### 1. 입력 받기
- 입력은 **마크다운이 가장 이상적**이지만, 그것만 받는 건 아니다. 폴더에 들어온 게 docx·pdf·pptx·txt·이미지 등이면 먼저 텍스트를 추출(markitdown/ocr 등)해 마크다운 수준으로 정규화한 뒤 진행한다. (경로를 인자로 받거나 사용자가 지정)
- 이미지가 있으면 마크다운 내 `![](상대경로)` 참조를 그대로 따른다.
- **인터뷰로 메운다**: 내용이 빈약하거나 구성 의도(슬라이드 수, 강조점, 표/차트로 뺄 것)가 불명확하면, 빌드 전에 사용자에게 **짧게 물어** 함께 다듬는다. 단, 묻는 건 *구성·강조·범위*에 한하고, **없는 사실·수치를 채워 넣기 위한 질문이 아니다**(충실도 철칙은 그대로). 답이 없으면 자동 판단으로 담백하게 간다.

### 2. 원문 추출 — 충실도 철칙 (위반 금지)
- **사실·수치·고유명사·인용·인과관계·순서는 추가/삭제/변형 0.** 원문에 없는 숫자·통계·출처·주장·예시를 **절대 지어내지 마라.**
- 허용: 슬라이드용으로 **문장 압축·제목화·재배열·요점화**. (의미 보존 하에)
- 모르는 건 비워라. 채우려고 환각하지 마라. 원문이 빈약하면 슬라이드도 담백하게.

### 3. 슬라이드 구성 (페이지네이션)
- 기본은 **완전 자동**: 의미 밀도에 따라 너가 알아서 나눈다. 한 슬라이드 = 한 메시지.
- **단, 마크다운에 명시된 PPT 지시는 무조건 준수**한다. 예: `<!-- slide -->` 구분, "이건 표로", "한 장에", 슬라이드 수 지정, 강조 표시 등 작성자의 정보량·구성 의도가 있으면 그대로 따른다.
- 표지(H1/문서 제목 기반)와 감사 페이지는 **자동 생성**된다(스펙의 `meta`로 넘기면 됨). 본문만 페이지 번호가 붙는다.

### 4. 시각화 매핑 — **데이터는 원문에 있는 것만**
원문 요소를 블록으로 매핑한다(상세 스키마: `design-system.md`). 차트·표·수치 카드는 **원문에 실제로 있는 표/숫자**로만 만든다. 값이 없으면 차트를 만들지 말고 텍스트 블록(bullets/numbered/callout)을 써라.
- 마크다운 표 → `table` (또는 수치 비교면 `bar_chart`/`column_chart`)
- 원문에 있는 시계열 수치 → `line_chart` / 구성비 → `donut_chart`
- 핵심 지표 숫자 묶음 → `stat_cards`
- 순서·단계 → `process`, 연/시점 시퀀스 → `timeline`
- 대비 구조(A vs B) → `comparison`, 일반 목록 → `bullets`, 번호 목록 → `numbered`
- 한 줄 핵심 → `callout`, 진척/비율(원문 % 있을 때) → `kpi_progress`
- 이미지 → 해당 위치 슬라이드에 이미지 블록(가능하면 카드/캡션과 함께)
밀도가 높아도 **디자인을 타협하지 말 것**: 한 슬라이드에 블록 1개 원칙, 넘치면 페이지를 나눈다.

### 5. 스펙 JSON 작성 → 빌드
- `decks/` 아래 다음 인덱스로 폴더를 만든다: `decks/NN-슬러그/` (예: `decks/02-반도체시장/`). NN은 기존 최대+1, 2자리.
- 원본 md(+이미지)도 그 폴더에 둔다(입력과 출력이 한 폴더에 모인다).
- 스펙을 `decks/NN-슬러그/spec.json`으로 저장(스키마: `design-system.md`).
- 빌드: `python scripts/build.py decks/NN-슬러그/spec.json --out decks/NN-슬러그/<슬러그>.pptx`
- 이미지는 build에서 상대경로로 참조되므로 spec의 이미지 경로는 그 폴더 기준 상대경로로 적는다.

### 6. 검증
- 빌드가 성공하면 `python -c "from pptx import Presentation; Presentation('...').slides"` 로 열림 확인.
- 의심되면 LibreOffice로 PDF 렌더해 육안 점검(겹침/넘침). 렌더러에 자동 축소(fit-to-box)가 있어 길이 초과는 대부분 자동 처리되지만, 표/차트 데이터 정확도는 원문과 대조한다.

## 서브에이전트 정책 (명세에 명문화 — 사소하게 띄우지 말 것)
- **기본은 단일 패스**(너 혼자 처리). 표지+본문이 **12장 이하**거나 단일 주제면 서브에이전트를 **띄우지 않는다.**
- **병렬은 큰 덱에서만**: 본문이 **13장 이상**이고 서로 독립적인 섹션이 여러 개일 때만, 섹션 묶음별로 콘텐츠 JSON 생성을 병렬화한다. 이때도 **렌더는 반드시 단일 `build.py`**로 모아 통일성을 지킨다.
- 사소한 변환·소규모 덱·단순 수정에 서브에이전트를 띄우는 것은 금지(오버헤드).

## 절대 규칙 요약
1. 사실/수치/고유명사 날조 금지. 원문에 있는 것만.
2. 디자인은 render.py가 단일 결정. 직접 그리지 말 것.
3. 차트/표는 원문 데이터로만. 없으면 텍스트 블록.
4. 한 슬라이드 한 메시지, 넘치면 분할(디자인 타협 금지).
5. 통일 방식(단일 렌더러)은 고정. 단, **디자인 자체는 사용자 피드백으로 계속 진화**시킨다(render.py를 다듬어 그 사용자 취향으로 발전).

## 파일
- `scripts/render.py` — 디자인 시스템(통일성의 단일 원천). 손대지 말 것(블록 추가 시 신중히).
- `scripts/build.py` — 스펙 JSON → pptx CLI.
- `scripts/build_template.py` — 스타일 베이스(assets/ppt-template.pptx) 재생성. 로고/accent 바꾸면 재실행.
- `assets/ppt-template.pptx` — 스타일 베이스(마스터/테마/레이아웃/배경/로고).
- `assets/your-logo.png` — 기본 로고(교체용).
- `config.json` — accent·logo·template·onboarded.
- `design-system.md` — 블록 스키마·매핑·예시.
- `decks/NN-슬러그/` — 주제별 입력+출력 한 폴더.
- `originals/` — 내부 보관(sample.pptx 예시 산출물, ppt-template-old.pptx 원본 아카이브).
